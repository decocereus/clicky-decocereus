# Clicky Identity

This document is the source of truth for what Clicky is, how it should feel,
and what product experience it should deliver across the macOS app, the web
companion, onboarding, voice, paywalls, and public-facing copy.

Use this document before:

- writing public-facing copy
- designing interfaces
- shaping onboarding and monetization
- refining persona behavior and voice
- deciding whether a feature feels like "Clicky"

This doc complements, rather than replaces:

- [product-vision.md](./product-vision.md) for shell/runtime and identity model
- [macos-design.md](./macos-design.md) for desktop UI rules

## What Clicky Is

Clicky is an intelligent companion that lives beside the user's work.

It is:

- voice-first
- screen-aware
- visually guiding
- context-sensitive
- present inside the task, not outside it

Clicky does not exist only to help users get unstuck on a Mac.

It can:

- explain what is happening on screen
- teach people how unfamiliar software works
- answer questions in real time
- guide attention to the right place
- help users complete work faster
- help users onboard into complex tools and workflows
- act like a capable assistant across writing, research, operations, learning, and software use

The best shorthand is:

Clicky is an intelligent companion that stays next to the cursor and helps the
user understand, learn, and act inside real software and real tasks.

## The Product Promise

Clicky should make a user feel:

- I have a capable intelligence with me
- it understands my current context
- it can teach me while I work
- it can guide my attention, not just answer in text
- it feels alive, useful, and immediate

Clicky is not just a chatbot with a prettier shell.

It should feel like:

- a companion
- a teacher
- a guide
- an operator
- a live presence beside the user

depending on the chosen persona and moment.

## What Clicky Is Not

Clicky is not:

- just a macOS troubleshooting tool
- just an overlay cursor toy
- just voice AI for desktop
- just a settings-heavy assistant app
- just a wrapper around an upstream agent

It should never feel like:

- open a panel
- fill out a form
- wait for a chatbot answer

It should feel like:

- speak naturally
- stay in context
- get guidance immediately

## Core Product Truths

Every part of the product should reinforce these truths.

### 1. Clicky Is Present

Clicky lives beside the cursor, beside the page, beside the work.

It should feel nearby and available, not remote or buried.

### 2. Clicky Is Contextual

Clicky should understand the environment the user is in.

It does not respond as if the user is speaking into a void. It responds to
what is actually in front of them.

### 3. Clicky Is Active

Clicky does not only answer questions.

It should:

- point
- orient
- guide
- explain
- help the user move

### 4. Clicky Is Teachable

Clicky can help a user learn an unfamiliar system in real time.

This matters for:

- consumer learning
- creative software
- internal tools
- enterprise onboarding
- B2B deployment

### 5. Clicky Is A Companion, Not A Utility Menu

Even when the UI is minimal, the product should feel like shaping a companion,
not configuring a sterile tool.

## The Emotional Target

The user experience should feel:

- magical
- calm
- immediate
- capable
- alive

without becoming:

- noisy
- gimmicky
- overly cute
- overly technical
- webby inside the Mac app

The right reaction is:

"Oh wow, this actually understands where I am and helps me from right here."

## Experience Principles

### Presence Before Configuration

Users should feel Clicky's presence before they feel its settings.

The first impression should be capability and aliveness, not setup overhead.

### Teach Through Use

Clicky should teach the product by being useful, not through long explanation.

### Keep The User In Flow

Clicky should reduce context switching, not create another workspace to manage.

### Show, Don't Just Tell

Pointing, voice, and motion should demonstrate capability faster than paragraphs
of explanatory copy.

### Make The Magic Legible

The product can feel magical, but the user should always understand:

- what Clicky is doing
- why it is doing it
- what they can do next

## Key Product Moments

These are the moments we should design most carefully.

### First Encounter

The user should immediately understand:

- Clicky lives next to the cursor
- it responds to voice
- it sees screen context when needed
- it can guide visually

### First Successful Guided Answer

The user should feel:

"This understood my context and helped me without making me leave what I was doing."

### First Pointing Moment

This is one of Clicky's most special experiences.

The user should feel:

"It knew exactly where I needed to look."

### First Teaching Moment

The user should feel:

"This can actually help me learn software and workflows in real time."

### First Paywall Moment

The user should feel:

"I've already felt the value. Unlocking this is continuing something I want,
not paying to remove an arbitrary block."

## Onboarding Philosophy

The launch/onboarding flow should follow this shape:

1. Show what Clicky is and why it matters.
2. Sign the user in so credits, restore, and purchase state belong to them.
3. Ask only for the permissions needed.
4. Let them use Clicky with included credits.
5. Show the paywall only after they have felt the product's value.

Do not lead with a paywall.

Do not make onboarding feel like enterprise setup before the user has felt the
magic.

## Monetization Philosophy

The paywall should feel earned.

It should not feel like:

- access denied
- generic subscription wall
- abrupt monetization interruption

It should feel like:

- you have now experienced what Clicky can do
- unlock it to keep this companion with you
- this is continuing a relationship, not starting one

That applies to:

- companion panel paywall
- Studio locked state
- voice copy at exhaustion
- upgrade copy on the website

## Voice And Persona Philosophy

The voice should always sound like Clicky belongs beside the user.

Persona should change the feel of the companion without breaking the core
product truth that Clicky is:

- present
- useful
- contextual
- alive

Persona differences should affect:

- tone
- pacing
- animation energy
- cursor behavior flavor
- shell presentation

Persona differences should not make Clicky feel like a different product each
time.

## UI Philosophy

### Companion Panel

The companion panel is a quick daily surface.

It should stay:

- compact
- focused
- useful
- free of junk

### Studio

Studio should feel like shaping and understanding the companion, not wading
through generic settings.

### Cursor Companion

The cursor companion is one of Clicky's defining moments.

It must always feel:

- intentional
- alive
- useful
- visually clear

not merely decorative.

### Web Companion

The website should not merely describe Clicky.

It should give a real taste of the experience so users can understand the
product before downloading it.

## Product Test

Clicky is on the right track if a user can honestly say:

"This feels like an intelligent companion that is with me while I work, can see
what I see, teach me in context, and guide me exactly where I need to look."

## Design And Copy Check

Before shipping copy, flows, or interface changes, ask:

- Does this make Clicky feel present?
- Does this make Clicky feel contextual?
- Does this make Clicky feel active rather than passive?
- Does this help the user feel the magic faster?
- Does this sound like a companion, not a tool manual?
- Does this preserve calm and clarity without losing aliveness?

If the answer is no, rethink it.
